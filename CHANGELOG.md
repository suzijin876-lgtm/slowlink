# 更新日志

本项目遵循语义化版本号。发布更新日志由本文件对应版本章节自动生成。

## [1.38.78] - 2026-07-12

- 支持 `Register/Renew` 随机后缀中的字面量 `数字`、`字母` 遮罩标记，避免完整码被截断或无法触发。
- 多行遮罩注册码会为每一行生成独立文本去重指纹，不再因漏提取而错误合并。
- 自动迁移 Redis 中旧的 ASCII 纯码触发规则和码识别规则，无需手工修改正则。
- 未知中文后缀不会被截成半个邀请码参与去重，继续避免宽泛中文误识别。

## [1.38.77] - 2026-07-11

- 发布安全测试改为检查敏感运行文件未被 Git 跟踪且被 `.gitignore` 忽略，不再要求删除本地 `.env`、Session、运行数据或 watchdog 日志。
- 补充 `secrets/`、Token、私钥、证书和 `credentials.json` 的忽略规则。
- 发布回归测试改为从 `VERSION` 读取当前版本，避免正常发版时重复修改历史测试文件。
- 整理历史 GitHub Release：移除线上更新日志附件，使用完整中文更新正文，并让 SHA-256 只校验两个 ZIP 资产。

## [1.38.76] - 2026-07-11

- 将 `SIGUSR1` Python 全线程调用栈从 Docker 标准日志隔离到容器内临时诊断文件，再由 watchdog 写入宿主机 `watchdog.log`。
- 每轮高 CPU 只在首次超限时采集一次调用栈，持续到自动重启时不再重复打印。
- watchdog 不再复制应用容器最近 160 行日志，避免调用栈在诊断日志中累积重复。
- 保留持续高 CPU 自动重启、Redis 状态、Telegram Session 和监听自动恢复行为。

## [1.38.75] - 2026-07-11

- 修复 Release 解压后源码时间固定且文件大小相同时，Docker BuildKit 可能复用旧构建上下文，导致宿主已更新但容器仍运行旧版本的问题。
- 部署前主动刷新 app 源码文件时间；健康检查后强制核对容器内版本与 `VERSION`。
- 版本不一致时自动执行一次 `slowlink_app` 无缓存重建并再次校验，整个过程仍不停止 Redis 或其他服务。

## [1.38.74] - 2026-07-11

- 监听入口增加动态 chat ID 快速过滤，未进入监听列表的消息在进入业务处理前直接跳过，降低大账号消息风暴时的单核 CPU 压力。
- 手工填写且暂时无法解析的监听项自动退回原有过滤路径，避免优化导致漏消息；监听配置修改后无需重启即可刷新过滤集合。
- 增加一分钟消息流诊断，记录收到、快速过滤、入队、命中、转发和队列数量，便于定位 Telegram 补发或群消息洪峰。
- 应用启用 SIGUSR1 Python 全线程调用栈输出；watchdog 在首次高 CPU 和持续高 CPU 时保存调用栈、消息流和线程快照。
- watchdog 不再记录主机进程完整命令参数，避免诊断日志意外包含其他服务的 Token 或密码。
- 保留原有持续高 CPU 自动重启、Redis 数据、Session 和监听自动恢复行为；不修改匹配、去重或转发业务逻辑。

## [1.38.73] - 2026-07-11

- 初始化公开 Git 仓库和 `main` 分支，清理仓库根目录历史版本包与运行时文件。
- 增加 GitHub Actions 标签发布流程，自动测试并生成 app 包、full 包、更新日志和 `SHA256SUMS.txt`。
- 增加中文一键安装、更新、卸载和管理脚本，支持 Ubuntu、Debian 与 `curl | sudo bash`。
- 更新时保留 `.env`、`data`、Telegram Session、Redis 数据和用户配置，只重建 `slowlink_app`。
- 保留现有 CPU watchdog 和 Redis 监听状态自动恢复逻辑。
- 本版本不修改消息监听、识别、去重或转发业务逻辑。
