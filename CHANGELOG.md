# 更新日志

本项目遵循语义化版本号。发布更新日志由本文件对应版本章节自动生成。

## [1.38.86] - 2026-07-15

- 精简 Telegram 高延迟轻量重连日志：一次重连只记录准备重连、转发结果和重连结果，不再重复打印开始、完成或失败状态。
- 重连原因不再附带原始正则片段，仅保留延迟、来源和连续次数等必要诊断信息。
- 高延迟重连事件统一通过网页事件通道写入，网页日志与 Docker 日志使用同一条文案。
- 连续 30 分钟未收到消息时改用简短中文提示，并以最后一条消息或最后一次重连为计时基准，避免安静频道反复重连；高延迟阈值、缓存复用和转发业务逻辑保持不变。

## [1.38.85] - 2026-07-15

- 网页后台“账号与转发 → 转发目标”新增公开链接格式切换，可选 `t.me 标准模式` 或 `telegram.me 兼容模式`。
- 新安装或尚未保存该设置时默认使用 `t.me`；非法或未知配置也会安全回退到 `t.me`。
- 保存后立即清理监听线程的设置缓存，后续新转发无需重启监听即可使用新域名。
- 私有群及私有话题链接始终保留 `https://t.me/c/...`，不受公开链接模式影响。
- 公开链接格式纳入配置备份与恢复，容器重建和服务器重启不会丢失用户选择。

## [1.38.84] - 2026-07-14

- 公开群和频道的帖子链接改用 `https://telegram.me/用户名/消息ID`，绕过当前 `t.me` 返回空网页预览的缓存路径。
- 公开论坛话题链接同步使用 `telegram.me`，保留话题 ID 和消息 ID 的原有结构。
- 私有群链接继续使用 `https://t.me/c/...`，不改变私有消息跳转兼容性。
- 识别、去重、监听和转发流程保持不变，本版本只调整公开消息链接的域名。

## [1.38.83] - 2026-07-14

- 修复同一场跨群抽奖在“无随机种子”和“带随机种子”两种模板中分别走全文与稳定 ID 去重、导致重复转发的问题。
- 高置信普通抽奖会根据创建者、参与关键词、开奖时间和规范化奖品清单生成 10 分钟跨模板关联指纹。
- 带抽奖 ID、随机种子和无稳定 ID 的模板都会检查同一关联指纹，并支持不同来源群的同一抽奖相互去重。
- 开奖时间或奖品清单发生变化时不会共享指纹；两个明确且不同的随机种子仍优先视为不同抽奖。
- 奖品指纹保留小数点，避免把 `1.5` 与 `15` 这类不同奖品数值误判为相同。

## [1.38.82] - 2026-07-14

- 修复“自由注册成功”个人账号通知被开放注册正则误判并转发的问题。
- 在所有关键词、正则和完整码触发前增加窄范围安全过滤：只有独立的“注册成功”状态同时带有创建账号、账号有效期或到期时间字段时才拦截。
- “自由注册已开启”等真实开放注册公告继续正常命中；“自由注册成功率提升”等包含相似字样的公告不会被误过滤。
- 网页正则测试会明确显示“个人注册成功通知过滤”，便于区分规则未命中与底层安全拦截。

## [1.38.81] - 2026-07-14

- 修复同一场无抽奖 ID 的刮刮乐使用“抽奖活动已开始”和“抽奖开始啦”两种模板时被重复转发的问题。
- 对能够明确提取来源、刮刮乐模式和至少两个币值奖品的消息，增加 10 分钟短时模板关联去重。
- 不同来源、不同奖品清单或超过 10 分钟后发布的新抽奖仍会正常转发，避免把后续正常活动误判为重复。
- 原有抽奖 ID、随机种子、规范化全文、邀请码和原消息链接去重逻辑保持不变。

## [1.38.80] - 2026-07-14

- 修复同一个抽奖以“参与关键词”和“参与条件”两种模板发布时被重复转发的问题。
- 抽奖消息现在优先提取明确的“抽奖 ID”或“随机种子哈希”作为稳定去重身份，不再把这些唯一字段丢弃后只比较正文。
- 相同稳定 ID 在抽奖去重时间内会记录为“相同抽奖 ID 重复”；不同 ID 即使创建者和奖品相同也会正常转发。
- 没有稳定 ID 的抽奖继续使用原有规范化文本去重，不启用容易误挡不同活动的模糊相似判断。

## [1.38.79] - 2026-07-12

- 支持强格式 `Register/Renew` 码后缀中用于遮挡的字面量 `*`，只在完整码结构内清理，不全局删除符号。
- `GuaiCum-30-Register_ZlB5*cqm*u` 会被完整提取为 `GuaiCum-30-Register_ZlB5cqmu`，避免截成半个码参与去重。
- 完整 `Register/Renew` 强格式码可以独立触发转发，不再依赖额外的主正则或“已为您生成”上下文。
- 继续过滤“注册码使用”等使用通知，也不会把含其它未知标点的后缀擅自拼接成邀请码。

## [1.38.78] - 2026-07-12

- 支持 `Register/Renew` 随机后缀中的字面量 `数字`、`字母` 遮罩标记，避免完整码被截断或无法触发。
- 多行遮罩注册码会为每一行生成独立文本去重指纹，不再因漏提取而错误合并。
- 自动迁移 Redis 中旧的 ASCII 纯码触发规则和码识别规则，无需手工修改正则。
- 未知中文后缀不会被截成半个邀请码参与去重，继续避免宽泛中文误识别。

## [1.38.77] - 2026-07-11

- 发布安全测试改为检查敏感运行文件未被 Git 跟踪且被 `.gitignore` 忽略，不再要求删除本地 `.env`、Session、运行数据或 watchdog 日志。
- 补充 `secrets/`、Token、私钥、证书和 `credentials.json` 的忽略规则。
- 发布回归测试改为从 `VERSION` 读取当前版本，避免正常发版时重复修改历史测试文件。
- 整理历史 GitHub Release：移除线上更新日志附件，使用完整中文更新正文，并让 SHA-256 只校验两个 ZIP 资产。

## [1.38.76] - 2026-07-11

- 将 `SIGUSR1` Python 全线程调用栈从 Docker 标准日志隔离到容器内临时诊断文件，再由 watchdog 写入宿主机 `watchdog.log`。
- 每轮高 CPU 只在首次超限时采集一次调用栈，持续到自动重启时不再重复打印。
- watchdog 不再复制应用容器最近 160 行日志，避免调用栈在诊断日志中累积重复。
- 保留持续高 CPU 自动重启、Redis 状态、Telegram Session 和监听自动恢复行为。

## [1.38.75] - 2026-07-11

- 修复 Release 解压后源码时间固定且文件大小相同时，Docker BuildKit 可能复用旧构建上下文，导致宿主已更新但容器仍运行旧版本的问题。
- 部署前主动刷新 app 源码文件时间；健康检查后强制核对容器内版本与 `VERSION`。
- 版本不一致时自动执行一次 `slowlink_app` 无缓存重建并再次校验，整个过程仍不停止 Redis 或其他服务。

## [1.38.74] - 2026-07-11

- 监听入口增加动态 chat ID 快速过滤，未进入监听列表的消息在进入业务处理前直接跳过，降低大账号消息风暴时的单核 CPU 压力。
- 手工填写且暂时无法解析的监听项自动退回原有过滤路径，避免优化导致漏消息；监听配置修改后无需重启即可刷新过滤集合。
- 增加一分钟消息流诊断，记录收到、快速过滤、入队、命中、转发和队列数量，便于定位 Telegram 补发或群消息洪峰。
- 应用启用 SIGUSR1 Python 全线程调用栈输出；watchdog 在首次高 CPU 和持续高 CPU 时保存调用栈、消息流和线程快照。
- watchdog 不再记录主机进程完整命令参数，避免诊断日志意外包含其他服务的 Token 或密码。
- 保留原有持续高 CPU 自动重启、Redis 数据、Session 和监听自动恢复行为；不修改匹配、去重或转发业务逻辑。

## [1.38.73] - 2026-07-11

- 初始化公开 Git 仓库和 `main` 分支，清理仓库根目录历史版本包与运行时文件。
- 增加 GitHub Actions 标签发布流程，自动测试并生成 app 包、full 包、更新日志和 `SHA256SUMS.txt`。
- 增加中文一键安装、更新、卸载和管理脚本，支持 Ubuntu、Debian 与 `curl | sudo bash`。
- 更新时保留 `.env`、`data`、Telegram Session、Redis 数据和用户配置，只重建 `slowlink_app`。
- 保留现有 CPU watchdog 和 Redis 监听状态自动恢复逻辑。
- 本版本不修改消息监听、识别、去重或转发业务逻辑。
